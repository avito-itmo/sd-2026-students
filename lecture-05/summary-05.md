# Лекция: Масштабируемость, отказоустойчивость и управление рисками

## Введение

 - **Контекст**: современное ПО работает в условиях высокой нагрузки, динамичного трафика и необходимости быть доступным
24/7

 - **Цель лекции**: понять, как проектировать архитектуры, способные масштабироваться, выдерживать сбои и управлять
качеством через метрики SLI/SLO/SLA

## Ключевые образовательные результаты:

 - Разрабатывать архитектуру для горизонтального и вертикального масштабирования

 - Применять стратегии обеспечения отказоустойчивости

 - Моделировать нагрузки и проектировать систему для пиковых режимов

 - Управлять качеством архитектуры через SLA/SLO/SLI

## 1. Масштабируемость: от одного узла к распределённой системе

### Типы масштабирования

 - Вертикальное (scale-up): увеличение мощности одного узла (ЦП, ОП, ДП)
   - Плюсы: просто, не требует изменения кода
   - Минусы: дорого, есть физический предел
 - Горизонтальное (scale-out): добавление новых узлов
   - Плюсы: практически бесконечное расширение
   - Минусы: требует изменения архитектуры (stateless, балансировка)

### Принципы горизонтального масштабирования
 - Stateless-компоненты: независимость экземпляров
 - Балансировка нагрузки: распределение трафика (Nginx, HAProxy, AWS ELB)
 - Масштабирование источников данных: репликация, шардирование, кеширование

### Масштабируемость источников данных

 - Шардирование: разделение данных на сегменты (по ID, гео-позиции и т. д.)
 - Репликация: master-slave, master-master
 - Eventual consistency: согласованность «в конечном счёте», trade-off между доступностью и согласованностью

## 2. Отказоустойчивость: проектируем на случай непредвиденного

### Классы сбоев
 - Аппаратные (отказ диска, сервера)
 - Сетевые (разрыв соединений, высокие задержки)
 - Программные (утечки памяти, баги)
 - Внешние зависимости (падение стороннего API)

### Паттерны отказоустойчивости
 - Retry: повтор запроса с задержкой (длительность может меняться)
 - Timeout: ограничение времени ожидания
 - Circuit breaker: разрыв цепи запросов при множественных сбоях
 - Fallback: упрощённый сценарий («дефолтные данные» вместо API)

### Репликация и кворум
 - Кворум: число реплик, участвующих в подтверждении операции
 - Баланс между консистентностью и доступностью

## 3. Моделирование поведения под нагрузкой

### Типы нагрузки
 - Регулярная (равномерный поток).
 - Пиковая (вечерние часы у стриминговых сервисов).
 - Burst (резкий скачок, флешмоб в соцсетях).
 - Непредсказуемая (DDOS, вирусное видео).

### Метрики нагрузки
 - RPS/QPS: запросы в секунду.
 - Latency: задержка.
 - Throughput: пропускная способность.
 - Queue depth: глубина очередей.

### Тестирование на нагрузку
 - Инструменты: JMeter, k6, Gatling.
 - Методики: стресс-тесты, spike-тесты, soak-тесты.

### Проектирование на пиковые сценарии
 - Adaptive scaling: автоматическое добавление узлов (Kubernetes HPA).
 - Graceful degradation: отключение «тяжёлых» функций, но сохранение базовой работы.

## 4. Архитектурное управление качеством через SLI/SLO/SLA

### Определения
  - SLA (Service Level Agreement): формальный контракт с бизнесом/клиентом.
  - SLO (Service Level Objective): целевые показатели внутри SLA.
  - SLI (Service Level Indicator): измеряемые метрики.

### Выбор SLI
  - Доступность (uptime).
  - Latency (p95/p99).
  - Ошибки (error rate).
  - Насыщенность (resource saturation).

### Определение SLO
  - Компромиссы: latency vs consistency, cost vs availability.
  - Пример кейса (онлайн-банк)
    - SLA: «доступность ≥ 99.9%».
    - SLO: «95% транзакций обрабатываются < 500 мс».
    - SLI: реальные измерения latency, error rate.

## Заключение и подведение итогов

 - Масштабируемость требует архитектурных изменений, а не только «железа».
 - Отказоустойчивость – это планирование сбоев и применение паттернов (retry, circuit breaker).
 - Нагрузочное тестирование помогает понять пределы системы до того, как они проявятся в продакшене.
 - SLI/SLO/SLA – это мост между бизнесом и архитектурой.
